## 🔒 Security & Git Tracking

### **✅ Tracked in Git** (Safe to commit)
- `docker/docker-compose.yml` - Docker container definitions
- `editor/*` - Editor configurations
- `environment/.webflowrc*` - Webflow CLI config (no secrets)
- `environment/env.example` - Environment template
- `Taskfile.yml` - Task definitions
- `boost-space-config.json` - Boost.space config
- `rensto-architecture.json` - Architecture documentation
- `cloudflared-oauth2-config.yml` - OAuth2 template (no real secrets)

### **❌ NOT Tracked (Ignored by .gitignore)**
- `docker/data/` - **Runtime data**: SQLite DBs, Redis dumps, logs (584K)
- `cloudflare-tunnel/` - **Tunnel credentials**: Contains tunnel IDs and auth tokens
- `environment/.env` - **Real secrets**: API keys, tokens, credentials
- `*credentials*.json` - **Any credentials files**: Airtable, QuickBooks, etc.
- `*.key`, `*.pem`, `*.crt` - **SSL/TLS files**

**⚠️ IMPORTANT**: Never commit files with real API keys or credentials. Use `.env.example` as a template.

## 📋 Configuration Files by Purpose

### **Docker & Containers** (`docker/`)

**docker-compose.yml**:
- Defines n8n, Redis, and PostgreSQL containers
- Used for local development
- **Usage**:
  ```bash
  cd configs/docker
  docker-compose up -d
  ```

**docker/data/** (NOT in git):
- Runtime data generated by Docker containers
- n8n SQLite database, Redis dumps, logs
- Created automatically on first `docker-compose up`
- **Backup**: Important! Contains n8n workflow data if using SQLite mode

### **Cloudflare Tunnel** (`cloudflare-tunnel/`)

**config.yml** (NOT in git):
- Cloudflare Tunnel ingress rules for n8n
- Routes `n8n.rensto.com` → `http://173.254.201.134:5678`
- Routes `n8n-oauth2.rensto.com` → OAuth2 callbacks
- **Setup**:
  ```bash
  cloudflared tunnel create rensto-n8n
  # Creates config in ~/.cloudflared/
  # Copy to configs/cloudflare-tunnel/config.yml
  ```

**cloudflared-oauth2-config.yml** (Template, tracked in git):
- Template for OAuth2 callback configuration
- Replace `YOUR-TUNNEL-ID-HERE` with actual tunnel ID

### **Editor Configuration** (`editor/`)

**.editorconfig**:
- Cross-editor coding standards
- Indentation, line endings, character encoding
- Supported by VS Code, Cursor, IntelliJ, etc.

**.prettierrc**:
- Code formatting rules for Prettier
- JavaScript/TypeScript formatting standards

### **Environment Variables** (`environment/`)

**.env** (NOT in git):
- Real API keys and secrets
- Webflow API token, database URLs, etc.
- **Created from**: Copy `env.example` and fill in real values

**env.example** (Template, tracked in git):
- Template showing required environment variables
- No real secrets, safe to commit
- **Usage**:
  ```bash
  cp environment/env.example environment/.env
  # Edit .env with real values
  ```

**.webflowrc** / **.webflowrc.json**:
- Webflow CLI configuration
- Site ID, API settings (no sensitive data)

### **Task Automation** (Root)

**Taskfile.yml**:
- Task automation using [Task](https://taskfile.dev)
- Define common development tasks
- **Usage**:
  ```bash
  task <task-name>
  ```

### **Integration Configs** (Root)

**boost-space-config.json** (12K):
- Boost.space integration configuration
- Workflow automation platform settings

**rensto-architecture.json**:
- System architecture documentation
- Defines services, integrations, data flows

### **Legacy Files** (Root)

**project-package.json** / **project-package-lock.json** (1.1M):
- ⚠️ **Potentially outdated**: These appear to be root-level package files
- **Action needed**: Verify if still needed or should be archived
- **Note**: Apps have their own package.json files

## 🔧 How to Use This Directory

### **For New Developers:**

1. **Set up environment variables**:
   ```bash
   cp configs/environment/env.example configs/environment/.env
   # Edit .env with real API keys (ask team lead)
   ```

2. **Start Docker services** (if using local n8n):
   ```bash
   cd configs/docker
   docker-compose up -d
   ```

3. **Configure editor** (optional):
   - Most editors auto-detect `.editorconfig` and `.prettierrc`
   - No manual setup needed

### **For Production Deployment:**

1. **Never commit secrets**: Always use environment variables or secret managers
2. **Use separate configs**: Production configs should be in deployment platform (Vercel, Railway, etc.)
3. **Docker data**: In production, use persistent volumes, not local `docker/data/`

### **For Adding New Configs:**

1. **Create subdirectory** if it's a category (e.g., `configs/postgres/`)
2. **Add to .gitignore** if it contains secrets
3. **Create `.example` template** for configs with secrets
4. **Document in this README**

## ⚠️ Common Issues

### **Issue: "My .env changes aren't showing up"**
**Solution**: `.env` is gitignored. This is correct - secrets should never be in git.

### **Issue: "Docker containers won't start"**
**Solution**: Check if `configs/docker/data/` has proper permissions:
```bash
chmod -R 755 configs/docker/data/
```

### **Issue: "Cloudflare tunnel not working"**
**Solution**:
1. Check tunnel credentials exist: `~/.cloudflared/your-tunnel-id.json`
2. Verify `config.yml` has correct tunnel ID
3. Test tunnel: `cloudflared tunnel run your-tunnel-name`

### **Issue: "Where do I put new config files?"**
**Decision Matrix**:
- **Application-specific config** → Put in app's directory (e.g., `apps/api/config/`)
- **System-wide config** → Put in `/configs/` with subdirectory
- **Secrets** → Use `.env` or secret manager, add to `.gitignore`
