**Worker job processing flow:**

1. **Job received** (OnboardingJobData with tenantId, groupId, clientPhone, triggeredBy)

2. **Initialize pipeline state:**
   - Call `fetchTenantProducts(tenantId)` to get tenant's products
   - Walk MODULE_REGISTRY (import from module-router.ts or replicate the product-matching logic) to determine which modules apply
   - Create pipeline state via `upsertPipelineState()` with available_modules list, status='active', admin_phone=triggeredBy

3. **Send module selection poll:**
   - Filter available_modules minus completed_modules to get remaining modules
   - If none remaining: go to completion flow (step 7)
   - Build poll options from remaining module types (human-readable labels: "Brand Assets", "Character Creation", "Character Video", "Social Media Setup", "Competitor Research")
   - Call `sendPoll(groupId, "Which module would you like to do next?", options)`
   - Update pipeline state: status='active', current_module=null (awaiting selection)

4. **IMPORTANT: The worker job ENDS here.** The poll response comes via WAHA webhook -> claudeclaw.worker.ts -> group-agent flow. The pipeline does NOT long-poll. Instead:
   - Add a new handler in the claudeclaw group message flow (or a helper function) that detects poll vote responses
   - When a poll vote is received for an onboarding group, map the selected option back to a ModuleType
   - Call `upsertPipelineState()` to set current_module
   - The existing module-router.ts `routeToModule()` already handles activating and running modules via message flow
   - When `routeToModule()` returns `{ completed: true }`, the caller should:
     a. Mark module as completed in pipeline state (add to completed_modules)
     b. Send celebration message (e.g., "Module complete! Great work.")
     c. After 30 seconds (use setTimeout -- per user's discretion), send next poll with remaining modules
     d. If current_module was last one, go to completion flow

5. **Module completion detection:**
   - In claudeclaw.worker.ts group message handling, after `routeToModule()` returns, check if `result.completed === true`
   - If completed: update pipeline state, notify admin "Module X complete for TenantName -- awaiting your approval to continue"
   - Set pipeline status to 'awaiting-approval'
   - Wait for admin APPROVE command (handled in Plan 02)

6. **Completion flow (all modules done):**
   - Send summary message to group listing all completed modules
   - Set pipeline status to 'awaiting-approval' for final admin approval
   - After admin approval: send "You're all set! Here's what's coming next for each product:" with per-product next steps
   - Set pipeline status to 'complete'

**Key implementation notes:**
- The worker is NOT a long-running process. It initializes the pipeline and sends the first poll. All subsequent interactions happen through the existing WAHA webhook -> claudeclaw message flow.
