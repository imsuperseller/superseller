    - Test: Duplicate request_id delivery does NOT trigger a second job update (idempotency)
    - Test: Error payload (status: "ERROR") updates job with error message
    - Test: Invalid/missing signature logs warning (when FAL_WEBHOOK_VERIFY=true)
  </behavior>
  <action>
Create `apps/worker/src/api/fal-webhook.ts`:

1. Export `falWebhookRouter = Router()`

2. Route: `POST /webhooks/fal`
   - Immediately respond `res.status(200).json({ status: 'ok' })` (fal.ai has 15s timeout)
   - After response, verify ED25519 signature if `process.env.FAL_WEBHOOK_VERIFY !== 'false'`:
     - Read headers: `x-fal-webhook-request-id`, `x-fal-webhook-timestamp`, `x-fal-webhook-signature`
     - Check timestamp within +/- 5 minutes
     - Construct message: `${requestId}\n${timestamp}\n${sha256(rawBody)}`
     - Fetch JWKS from `https://rest.fal.ai/.well-known/jwks.json` (cache in module-level variable for 24h)
     - Verify with `crypto.verify('ed25519', msgBuffer, key, sigBuffer)` against each key
     - If invalid: `logger.warn(...)` and return (do not process)
   - Gate: set `FAL_WEBHOOK_VERIFY=false` env var initially for testing (ED25519 exact message format needs live validation per RESEARCH open question #2)

3. Process completion:
   - Extract `request_id`, `status`, `payload` from body
   - Parse `request_id` to find the matching BullMQ job or DB record
   - For status 'OK': extract `payload.video.url`, update job record with resultUrl and status='complete'. Use idempotent SQL: `UPDATE ... WHERE fal_request_id = $1 AND status != 'complete'`
   - For status 'ERROR': update job record with error message
   - Call `trackExpense()` on successful completion for cost attribution

4. Mount in routes.ts: `import { falWebhookRouter } from './fal-webhook'; apiRouter.use(falWebhookRouter);`

5. Create `apps/worker/src/api/fal-webhook.test.ts` with mocked DB query and fetch. Test the 200 ack, idempotency, error handling, and signature rejection paths.

Note: The exact ED25519 message construction (whether userId is included, exact field order) should be verified against a real fal.ai job during the test job phase. The implementation should be gated behind `FAL_WEBHOOK_VERIFY` env var so the webhook works even if signature format needs adjustment.
