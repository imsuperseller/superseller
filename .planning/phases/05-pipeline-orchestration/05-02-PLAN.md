**1. Admin command handler in onboarding.worker.ts:**

Add `handleAdminCommand(groupId: string, command: string, adminPhone: string)` function:
- Parse command: APPROVE, RETRY, SKIP, PAUSE (case-insensitive)
- **APPROVE:**
  - If pipeline status is 'awaiting-approval' and current_module is set: mark module complete in completed_modules, update pipeline state
  - Send celebration to group: "Module [name] approved! Great progress."
  - After 30s delay (setTimeout), check remaining modules. If remaining: send poll. If none: set pipeline to 'awaiting-approval' for final approval with summary message
  - If this is the FINAL approval (all modules done, pipeline awaiting final): set status='complete', send "You're all set!" message with per-product next steps to the group
- **RETRY:** Re-activate current_module (reset its module state phase to 'intro'), set pipeline status='active'
- **SKIP:** Move current_module to completed_modules without running it, send next poll
- **PAUSE:** Set pipeline status='paused', send message to admin confirming pause

**2. Retry logic with failure handling:**

Add retry tracking in pipeline state (add `retry_counts` JSONB column to onboarding_pipeline -- update pipeline-state.ts table init):
```typescript
retry_counts: Record<string, number>  // { "asset-collection": 2, ... }
```

When a module fails (detected via routeToModule error or module state going to 'failed'):
- Increment retry count for that module
- If retry_count < 3: reset module state to 'intro', wait 30s, re-activate
- If retry_count >= 3: pause module, set pipeline status='paused', alert admin via WhatsApp:
  ```
  Pipeline Alert: Module [name] failed 3 times for [tenant]

  Reply:
  RETRY - Try again
  SKIP - Skip this module
  PAUSE - Pause pipeline
  ```
- Failures invisible to customer: if customer sends a message while paused, respond "Working on it -- we're making sure everything is perfect for you!"

**3. Stale detection:**

Add a periodic check (call it from a simple setInterval in the worker file, every 6 hours):
- Query `getAllActivePipelines()` where status != 'complete' and status != 'failed'
- For each: check `updated_at`
  - If > 48h since last update and status is 'active': send nudge to group ("Hey! Ready to continue? Let me know which module you'd like to work on next.")
  - If > 7 days since last update: alert admin via WhatsApp ("Pipeline stale for [tenant] -- no activity in 7 days")
- Update pipeline state timestamp after nudge/alert to avoid spamming

**4. Admin WhatsApp notifications:**

Create helper `notifyAdmin(adminPhone: string, message: string)` that calls `sendText(phoneToChatId(adminPhone), message)`.

Notification triggers (call from handlePipelineEvent and handleAdminCommand):
- Pipeline started: "Onboarding started for [tenant]. [N] modules to complete."
- Module approval needed: "Module [name] complete for [tenant]. Reply APPROVE to continue."
- Failure alert: (see retry section above)
- All modules complete: "All [N] modules complete for [tenant]. Reply APPROVE for final sign-off."

**5. Cost aggregation per module:**

After each module completes, query `api_expenses` table for costs associated with this tenant's pipeline run:
```sql
SELECT SUM(estimated_cost) as total FROM api_expenses WHERE job_id = $pipelineRunId
```
Update pipeline state `total_cost_cents` with sum. Also store per-module cost in collected_data or a separate JSONB field `module_costs` on onboarding_pipeline table.

**6. Wire into claudeclaw.worker.ts:**

In the group message handling section of claudeclaw.worker.ts:
- After `routeToModule()` returns, check `result.completed === true`
- If completed: call `handlePipelineEvent(groupId, tenantId, { type: 'module-completed', moduleType: result.moduleType })`
- Detect admin messages (from admin phone matching pipeline state's admin_phone): if message matches APPROVE/RETRY/SKIP/PAUSE pattern, call `handleAdminCommand(groupId, command, senderPhone)`
- Detect poll vote messages: WAHA sends poll votes as webhook events. Check if `poll.vote` event type is received, map selected option to module type, call `handlePipelineEvent(groupId, tenantId, { type: 'module-selected', moduleType })`

Note: Check how WAHA delivers poll votes in the webhook. It may come as a regular message with the selected option text, or as a separate `poll.vote` event. Handle both cases.
