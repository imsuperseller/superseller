```prisma
model WebhookEvent {
  id          String   @id @default(dbgenerated("gen_random_uuid()")) @db.Uuid
  provider    String   // "paypal" or "stripe"
  eventId     String   // Provider's event ID (event.id for Stripe, event.id for PayPal)
  eventType   String   // e.g., "checkout.session.completed", "BILLING.SUBSCRIPTION.ACTIVATED"
  status      String   @default("processing") // "processing", "completed", "failed"
  payload     Json?    // Raw event payload for debugging
  errorMessage String?
  retryCount  Int      @default(0)
  processedAt DateTime?
  createdAt   DateTime @default(now())
  updatedAt   DateTime @default(now()) @updatedAt

  @@unique([provider, eventId])
  @@index([provider, eventType])
  @@index([status])
}
```

After adding the model, run `cd apps/web/superseller-site && npx prisma db push` to apply. Then `npx prisma generate` to update the client.

**Part A — ProvisioningService.onboardNewCustomer():**

Add a new static method to ProvisioningService that both PayPal and Stripe webhooks will call. This method handles the full post-payment onboarding chain:

```typescript
static async onboardNewCustomer(params: {
  provider: 'paypal' | 'stripe';
  eventId: string;
  eventType: string;
  payload: any;
  customerEmail: string;
  customerName: string;
  customerPhone: string;
  productName: string;
  serviceType: string;
  subscriptionId: string;
  amount: number;
  metadata?: Record<string, any>;
}): Promise<{ tenantId: string; serviceInstanceId: string }> {
```

Implementation steps inside the method:

1. **Idempotency check**: Try to create WebhookEvent record with (provider, eventId). If unique violation (P2002), log and return early — already processed. Use try/catch on prisma.webhookEvent.create.

2. **Find or create Tenant**: Check if a Tenant already exists for this email (look up User by email, then TenantUser). If exists, use that tenant (multi-product case per decisions). If not, create new Tenant with:
   - name: params.customerName's Business (or derive from customerName)
   - slug: derive from customerName (slugify, ensure unique by appending random suffix if conflict)
   - status: 'active'
   - settings: { paymentProvider: params.provider, subscriptionId: params.subscriptionId }

3. **Get or create User + TenantUser**: Use existing `getOrCreateUser()`, then upsert TenantUser with role 'owner'.

4. **Create ServiceInstance**: Using prisma.serviceInstance.create with productName, serviceType, tenantId, status 'pending_setup'.

5. **Call worker to start onboarding**: HTTP POST to `${WORKER_API_URL}/api/onboarding/start` with body `{ tenantId, clientPhone: params.customerPhone, triggeredBy: 'webhook:' + params.provider, productName: params.productName, serviceType: params.serviceType }`. Include header `x-worker-secret: ${process.env.WORKER_API_SECRET}`. Use the existing env var WORKER_URL or fall back to `http://172.245.56.50:3002`.

6. **Send rich WhatsApp notification to Shai** (14695885133@c.us): Include customer name, amount, product, phone, and the onboarding group link (from worker response). Use the same sendWhatsApp pattern from the Stripe webhook (WAHA API call).

7. **Update WebhookEvent status**: Set status='completed', processedAt=new Date().

8. **On failure**: Wrap in try/catch. On error, update WebhookEvent status='failed', errorMessage. If retryCount < 3, throw to allow BullMQ retry. If retryCount >= 3, send WhatsApp alert to Shai with failure details + customer email for manual follow-up.

Important: The worker call is fire-and-forget with a 10-second timeout. If worker is unreachable, the WebhookEvent stays in 'processing' status for manual retry.

Import prisma at top. Use existing import pattern from the file.
