**Part A: Create `change-request-handler.ts`**

Create `apps/worker/src/services/onboarding/change-request-handler.ts` — the orchestrator that ties together classification, cost estimation, and poll confirmation.

1. **`handleChangeRequest(params)`** — Main entry point called when a customer sends a message in `delivered` phase. Params:
   ```typescript
   {
     groupId: string;
     tenantId: string;
     messageBody: string;
     state: ModuleState;  // from module-state.ts
   }
   ```

   Flow:
   a. Fetch CharacterBible via `getLatestCharacterBible(tenantId)` — need characterName and scene labels
   b. Build sceneLabels from bible.metadata (reuse logic from character-video-gen.ts `buildSceneLabels` — import or duplicate the 5-label array logic: `["Studio Portrait", scenarioNames[0] ?? "Brand in Action", ...]`)
   c. Call `classifyChangeRequest({ messageBody, sceneLabels, characterName: bible.name })`
   d. Create change request record via `createChangeRequest({ groupId, tenantId, messageBody, intent: classification.intent, sceneNumber: classification.sceneNumber, changeDescription: classification.changeDescription })`
   e. Route based on intent:

   - **`scene-change`**: Calculate cost estimate. Scene regen cost = `COST_RATES.fal.sora_2_per_second_1080p * 10` (10 seconds per scene = $5.00) + Remotion render ($0) = 500 cents. Update change request with costEstimateCents. Send cost confirmation poll via `sendPoll(groupId, "Scene regen will use $5.00 in credits. Proceed?", ["Yes, regenerate", "No, keep current"])`. Store `{ changeRequestId: cr.id, pollType: 'credit-confirmation' }` in module collectedData via upsertModuleState. Update change request status to `awaiting-confirmation`.

   - **`character-change`**: Calculate multi-scene cost estimate. Count affected scenes (default: all 5). Cost = 5 * 500 = 2500 cents ($25.00). Send poll: "Character change affects all 5 scenes. Estimated cost: $25.00 in credits. Proceed?", ["Yes, proceed", "No, cancel"]. Store changeRequestId in collectedData. Update status to `awaiting-confirmation`. Note: Actual character-level regen is Phase 18 — this phase only captures and confirms intent.

   - **`positive-feedback`**: Send `sendText(groupId, "Thank you! Glad you like it! If you'd like any changes later, just let me know here.")`. Update status to `completed`. Return `{ handled: true, response: null }` (response already sent).

   - **`unrelated`**: Return `{ handled: false }` — let it fall through to normal ClaudeClaw handling.

   - **`ambiguous`**: Send `sendText(groupId, classification.clarifyingQuestion)`. Update status to `classified`. Return `{ handled: true, response: null }`.

   Return type: `{ handled: boolean; response: string | null }` (consistent with ModuleHandleResult pattern but simplified).

2. **`handleChangeRequestPollVote(params)`** — Called when customer votes on credit confirmation poll. Params:
   ```typescript
   {
     groupId: string;
     tenantId: string;
     selectedOption: string;
     collectedData: Record<string, any>;  // contains changeRequestId
   }
   ```

   Flow:
   a. Get changeRequestId from `collectedData.changeRequestId`
   b. If selectedOption starts with "Yes":
      - Update change request status to `confirmed`
      - Send `sendText(groupId, "Confirmed! Your change request has been queued. Generation will begin shortly.")` (Note: actual generation dispatch is Phase 17/18 — this phase only confirms)
      - Return `{ handled: true }`
   c. If selectedOption starts with "No":
      - Update change request status to `rejected`
      - Send `sendText(groupId, "No problem! Your current video stays as is. Let me know if you'd like anything else.")`
      - Return `{ handled: true }`

3. **Cost estimation helper** (private): `estimateRegenCost(intent, sceneCount)`:
   - `scene-change`: `Math.round(COST_RATES.fal.sora_2_per_second_1080p * 10 * 100)` = 500 cents per scene
   - `character-change`: `sceneCount * 500` cents (all scenes)
   - Returns cents as integer

Imports: `classifyChangeRequest` from `./intent-classifier`, `createChangeRequest`/`updateChangeRequestStatus` from `./change-request-intake`, `getLatestCharacterBible` from `./character-bible-versioning`, `sendText`/`sendPoll` from `../waha-client`, `COST_RATES` from `../expense-tracker`, `upsertModuleState` from `./module-state`, `logger` from `../../utils/logger`.

**Part B: Update `claudeclaw.worker.ts` — Poll vote routing for change request polls**

The existing poll vote detection block (lines 206-223) routes ALL poll votes in onboarding groups to `handlePipelineEvent({ type: "poll-vote", ... })`. Change request credit confirmation polls need a different handler.

Add a **new poll vote detection block** for change request polls, BEFORE the existing pipeline poll vote block (insert before line 206):

```typescript
// ─── Change Request: Credit Confirmation Poll Vote ───
try {
    const groupCfg = getGroupConfig(chatId);
    if (groupCfg?.tenantId && job.data.isPollVote && job.data.pollOption) {
        const { getModuleState } = await import("../../services/onboarding/module-state");
        const moduleState = await getModuleState(chatId, "character-video-gen");

        if (moduleState?.phase === "delivered" && moduleState.collectedData?.changeRequestId) {
            const { handleChangeRequestPollVote } = await import("../../services/onboarding/change-request-handler");
            await handleChangeRequestPollVote({
                groupId: chatId,
                tenantId: groupCfg.tenantId,
                selectedOption: job.data.pollOption,
                collectedData: moduleState.collectedData,
            });
            return { handled: "change-request-poll-vote", isGroup: true };
        }
    }
} catch (err: any) {
    logger.warn({ msg: "Change request poll vote handling error (non-critical)", error: err.message });
}
```

This block checks:
1. Is it a poll vote in a group? (isPollVote && pollOption)
2. Is the character-video-gen module in `delivered` phase? (moduleState.phase === "delivered")
3. Does collectedData have a changeRequestId? (means a cost confirmation poll was sent)

If all true, route to `handleChangeRequestPollVote`. Otherwise, fall through to the existing pipeline poll vote handler.
