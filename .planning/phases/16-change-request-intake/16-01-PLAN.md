   export interface ChangeRequestClassification {
     intent: 'scene-change' | 'character-change' | 'positive-feedback' | 'unrelated' | 'ambiguous';
     sceneNumber: number | null;       // 1-5, extracted from message
     changeDescription: string | null; // what the customer wants changed

2. **`classifyChangeRequest(params)`** — Single Claude API call that classifies intent AND extracts scene number. Params:
   ```typescript
   { messageBody: string; sceneLabels: string[]; characterName: string }
   ```

   Implementation:
   - Use `fetch("https://api.anthropic.com/v1/messages", ...)` with `config.anthropic.apiKey` (same pattern as claude-bridge.ts)
   - Model: `claude-sonnet-4-20250514` (fast, cheap, sufficient for classification)
   - System prompt instructs Claude to output JSON with the ChangeRequestClassification schema
   - System prompt includes the 5 scene labels so Claude can match "the coffee shop scene" to a scene number
   - System prompt lists the 4 intent categories with examples:
     - `scene-change`: "make her hair shorter in scene 3", "change the lighting in the studio shot"
     - `character-change`: "make her more casual", "change the outfit to business formal", "rename to Sarah"
     - `positive-feedback`: "love it!", "this is amazing", "perfect"
     - `unrelated`: anything not about the character/video
     - `ambiguous`: unclear what they want changed — generate a clarifying question
   - Parse JSON from Claude response. If JSON parsing fails, return `{ intent: 'ambiguous', sceneNumber: null, changeDescription: null, clarifyingQuestion: 'Could you describe what you would like changed?', confidence: 0 }`

   - Track cost via `trackExpense({ service: 'anthropic', operation: 'intent-classification', estimatedCost: 0.005 })`

3. **Error handling**: Wrap fetch in try/catch. On API failure (network, 5xx), log error and return ambiguous classification with clarifyingQuestion "I'm having trouble understanding right now. Could you try again?" — never throw.

Create `apps/worker/src/services/onboarding/character-bible-versioning.ts` with:

1. **`getLatestCharacterBible(tenantId)`** — Same query as `fetchCharacterBible` in character-video-gen.ts but exported for reuse:
   ```sql
   SELECT id, "tenantId", name, "personaDescription", "visualStyle", "soraHandle",
          "referenceVideoUrls", "testSceneUrls", metadata, version, active, "createdAt"
   FROM "CharacterBible"
   WHERE "tenantId" = $1
   ORDER BY "createdAt" DESC
   LIMIT 1
   ```
   Return type includes `version: number` field.

2. **`createCharacterBibleVersion(params)`** — Versioned insert per PROJECT.md decision. Params:
   ```typescript
   {
     tenantId: string;
     changes: Record<string, any>;  // fields to change (e.g., { name: "New Name", visualStyle: "..." })
     changeDelta: Record<string, { from: any; to: any }>; // diff showing what changed
   }
   ```
   Implementation:
   - Fetch latest version with `getLatestCharacterBible(tenantId)`
   - If no existing bible, throw Error("No CharacterBible found for tenant")
   - Merge existing fields with `changes` (changes override existing values)
   - INSERT new row with:
     - New cuid() id (use `gen_random_uuid()::text` since worker uses raw SQL)
     - Same tenantId
     - Merged field values
     - `version = previousVersion + 1`
     - `active = true`
     - `metadata` merged with `{ changeDelta, previousVersionId: existingBible.id }`
   - Set previous version `active = false`: `UPDATE "CharacterBible" SET active = false WHERE id = $1`
   - Return the new row

3. **`getCharacterBibleVersion(tenantId, version)`** — Fetch specific version:
   ```sql
   SELECT * FROM "CharacterBible" WHERE "tenantId" = $1 AND version = $2
   ```

Import `query`/`queryOne` from `../../db/client`, `logger` from `../../utils/logger`.

Note: The CharacterBible table already has `version Int @default(1)` and `active Boolean @default(true)` columns in the Prisma schema, so NO migration needed. The `changeDelta` is stored inside the existing `metadata` JSONB column (as `metadata.changeDelta`), NOT as a separate column.
