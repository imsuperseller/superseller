## The video generation pipeline is technically proven
The most critical validation: **Veo 3.1 fully supports both start-frame and end-frame specification** via its `last_frame` parameter. This confirms the core continuity-chain concept works — each clip's end frame becomes the next clip's start frame, creating seamless room-to-room transitions. The model is publicly available through the Gemini API and Vertex AI in paid preview, with model IDs `veo-3.1-generate-001` and `veo-3.1-generate-preview`.
Veo 3.1 also supports **up to 3–4 reference images** ("Ingredients to Video") that preserve a subject's appearance throughout generation. Google's February 2026 documentation states identity consistency is "better than ever" with this feature. Combined with frame chaining, this addresses both motion continuity and character persistence — the two hardest problems in multi-clip AI video.

## Floorplan intelligence relies on vision LLMs, not specialized APIs
A peer-reviewed study (arXiv:2409.12842) found that **GPT-4o achieved a 0.96 success rate** on navigation planning tasks across floorplans with 9–11 rooms and dense labels. Both GPT-4o and Claude successfully identified rooms, doors, spatial connections, and generated optimal tour paths in 132 out of 150 trials. This makes vision LLMs the fastest, cheapest approach for determining room sequence — no specialized API needed.
The recommended pipeline sends the floorplan image to GPT-4o or Gemini 2.5 Pro via **OpenRouter** ($0.01–0.05 per analysis) with a structured prompt requesting JSON output: room names, approximate positions, door connections, and a suggested tour sequence. OpenRouter provides access to **300+ models** through a single OpenAI-compatible API at `https://openrouter.ai/api/v1`, with automatic fallback across providers if one is down. During MVP, OpenRouter's flexibility for switching between models with a single line change outweighs the small markup over direct API access.
**Important accuracy caveats**: performance degrades significantly on large floorplans (20+ rooms) and those with sparse labels. Complex L-shaped rooms and multi-story layouts reduce accuracy. The SaaS should allow agents to **manually override the AI-generated tour sequence** — reordering, adding, or removing rooms before generation begins.
**Specialized alternatives exist but are overkill for this use case.** Archilogic offers full floorplan digitization with a Space GraphQL API for querying room relationships, but it's enterprise-priced and designed for commercial space management. CubiCasa is a scanning app, not an image parser. For the specific need of "determine room order for a tour video," a $0.03 LLM call beats a $50+ API subscription.
**Zillow integration is effectively dead for developers.** The old ZWSID free API is deprecated, current APIs require partnership approval that takes weeks to months, and there is no endpoint for floorplan images specifically. Scraping violates Zillow's ToS and carries real legal risk — the *VHT, Inc. v. Zillow* case confirmed copyright liability for using photos beyond licensed scope. **The correct MVP approach is agent-uploaded content only.** For Phase 2, Bridge Interactive and SimplyRETS offer MLS data access through RESO Web API standards, but each MLS requires its own licensing agreement.

## AITable.ai must be replaced — the rest of the stack works
**AITable.ai is fundamentally unsuitable as a production SaaS database.** It's a spreadsheet/project management tool (an Airtable clone) with a **10,000-row-per-datasheet limit**, a beta API, 1 GB attachment cap per space, no transactions, no SQL queries, no row-level security, and no relational integrity. A video generation SaaS needs high write throughput for job queues, transactional consistency for billing, and complex queries across users, jobs, and subscriptions. AITable fails every requirement.
**Replace with Supabase** ($25/month Pro plan), which provides PostgreSQL, built-in auth (50K MAUs free), file storage, real-time subscriptions, and row-level security in one platform. The database schema needs tables for users, teams, listings, floorplans, video jobs, individual clips, final videos, subscriptions, and usage tracking.
The remaining stack is well-architected:
**Vercel** ($20/month Pro) handles the Next.js frontend with SSR, API routes for lightweight operations, and automatic preview deployments per PR. Set a **$200/month spend limit** to prevent overage surprises. Vercel's zero-config GitHub integration eliminates CI/CD complexity for the frontend.
**RackNerd** requires proper sizing — the promotional $11/year VPS is entirely insufficient for video processing. Budget a **dedicated server at $50–100/month** minimum (4+ cores, 16GB+ RAM) for FFmpeg workloads. This VPS runs the video processing pipeline: downloading clips from storage, normalizing resolution/framerate, concatenating with transitions via FFmpeg's `xfade` filter, overlaying background music, generating multi-format outputs, and uploading to R2. Deploy with PM2 for process management and BullMQ + Redis for job queue management.
**Cloudflare R2** is the ideal video storage solution with **zero egress fees** — storing 1,000 videos (~75 GB) costs roughly $1.13/month with unlimited delivery through Cloudflare's CDN. Compare this to AWS S3 where the same delivery could cost $50+/month in egress alone. R2's S3-compatible API means standard AWS SDK tools work without modification.
For **authentication**, Clerk offers the best Next.js developer experience (setup in minutes, pre-built components, 10K MAUs free) while Supabase Auth provides better value at scale (50K MAUs free, bundled with the database). The optimal combo: **Clerk for auth UX + Supabase for database**, with Clerk JWTs securing Supabase Row-Level Security policies. This is a well-documented integration pattern.
**Stripe** handles payments through its Meter entity (introduced March 2024), purpose-built for usage-based billing. The architecture: create a flat subscription price ($99/month for 15 videos) plus a metered overage price ($5/video beyond the included amount). When a video generates successfully, fire a `stripe.billing.meterEvents.create()` call. Stripe calculates overages automatically at billing cycle end. Transaction fees are 2.9% + $0.30 plus 0.5% for Billing.
The complete infrastructure costs approximately **$100–185/month** before API usage, which is remarkably lean for a production SaaS.

## Video stitching, music, and delivery form a proven pipeline
FFmpeg's **`xfade` filter** is the recommended approach for concatenating clips with smooth transitions — it supports 79+ transition types (dissolve, fade, wipeleft, circleopen, radial) and pairs with `acrossfade` for audio. For 10–15 clips, a Python script dynamically builds the filter graph, calculating offsets for each clip pair. Processing time for a 90-second final video at 1080p: **30–90 seconds** on a server with 4+ cores.
For faster time-to-market, **Shotstack** ($49/month for 200 minutes at 720p) handles stitching, transitions, audio overlay, and multi-format rendering in a single JSON API call — eliminating server-side FFmpeg management entirely. Creatomate offers similar capabilities at $41/month. Either cloud API is recommended for MVP, migrating to self-hosted FFmpeg when cost optimization matters at scale.
**Background music licensing for SaaS requires redistribution rights**, not standard creator licenses. Two approaches work:
- **Epidemic Sound Partner API**: 50,000+ tracks with full commercial clearance including sync, mechanical, and public performance rights. Requires partnership agreement with custom pricing. API supports search, curation, and programmatic download — purpose-built for platform integration.
- **Mubert API**: AI-generated ambient music at $99/month for 1,000 generations. Infinite generative tracks with 100+ mood presets. Lower cost, commercial license included, excellent for real estate ambient backgrounds.
Standard "royalty-free" subscriptions from Artlist or Musicbed typically do **not** cover SaaS redistribution. Bensound specifically warns against embedding music in SaaS products without enterprise licensing. Budget **$0.50–2.00 per video** for music amortization.
The delivery pipeline: generate a **1080p 16:9 master** video (H.264, CRF 18, AAC 192kbps, `-movflags +faststart`), then programmatically crop to 9:16 vertical (TikTok/Reels), 1:1 square (Instagram feed), and 4:5 portrait variants. FFmpeg handles all four outputs in a single pass. Store all variants on Cloudflare R2, serve through Cloudflare CDN, and generate 3–4 thumbnails per video using FFmpeg's scene detection.

## Identity consistency is solvable but legally complex
Maintaining a realtor's appearance across 10+ AI-generated clips is technically achievable through Veo 3.1's reference image system. The workflow: collect **3 reference photos** of the realtor (front, 3/4 view, side), pass them as `reference_images` in each API call alongside the start/end frames, and use consistent templated prompts with identical character descriptions. Google's documentation confirms this preserves subject appearance even as settings change.
However, **the legal landscape demands extreme caution.** California's AB 723 (effective January 1, 2026) requires agents to include "reasonably conspicuous" disclosure when listing content has been AI-modified, plus a link to original unaltered images. Tennessee's ELVIS Act specifically prohibits AI mimicry of a person's likeness without consent. The federal NO FAKES Act (pending) would make unauthorized AI likenesses unlawful nationally. Pennsylvania and Washington have already criminalized certain deepfake uses.
**The practical requirement is explicit written consent** — a signed digital release from each realtor covering the specific use case, AI generation method, distribution platforms, and usage duration. Because the realtor is the customer actively requesting their likeness be used, consent is straightforward to obtain but **must be documented** in the product's onboarding flow.
**Strategic recommendation: defer realtor-in-video to V2.** The most legally complex and technically challenging feature should wait until consent frameworks are production-tested. **V1 should generate cinematic empty walkthrough videos** — a camera smoothly moving through each room without a person visible. This is still enormously valuable: it's 10–50× cheaper than traditional video production ($200–900 for a basic agent-led tour) and eliminates likeness liability entirely. The walking-realtor feature becomes a premium V2 upgrade once the core pipeline is battle-tested.

## What's missing: 12 critical gaps in the current concept
The app concept has several important gaps that need addressing before development:
**Architectural gaps** that must be solved for MVP: there is no quality control or approval workflow — agents need to preview and approve/reject individual clips before final assembly, with one-click regeneration for rejected clips. The concept has no fallback for when no floorplan exists (many listings lack one) — build a manual room-ordering interface where agents drag-and-drop rooms into tour sequence. There is no handling for exterior shots, backyard, or garage — floorplans don't include these, but curb appeal and outdoor spaces are critical to real estate marketing. The concept also lacks processing time management — at 5–20 minutes per tour, the UX must be fully asynchronous with email/push notifications on completion.
**Technical edge cases** for V2 consideration: multi-story homes require detecting multiple floors in the floorplan and generating convincing staircase transition clips. Open-concept layouts need different handling than discrete rooms — the AI must treat large open spaces as single scenes rather than separate rooms. Floorplan-photo mismatches (floorplan labels a room as bedroom but the photo shows a kitchen) need a validation step that cross-references AI room identification against floorplan labels. Furnished versus unfurnished properties produce dramatically different video quality — virtually staged photos as input compound AI hallucination issues.
**Scalability concerns**: Veo preview quotas impose per-minute and per-day rate limits. At 100+ concurrent users, queue management with priority tiers, rate limiting, and exponential backoff becomes critical. The RackNerd VPS doesn't auto-scale — plan for manual scaling or migration to auto-scaling infrastructure when usage patterns become clear.

## Conclusion: a viable product with one killer insight
The technical foundation is solid — Veo 3.1's start-plus-end-frame control combined with reference images makes the continuity-chain video pipeline work today, not theoretically. The real insight is **strategic sequencing**: launch V1 as cinematic empty walkthroughs (legally clean, technically simpler, still 10× cheaper than traditional video), use that to validate product-market fit and refine the pipeline, then layer in realtor-likeness generation as a premium V2 feature once consent frameworks and identity consistency are production-hardened.
The three highest-priority actions are: first, replace AITable.ai with Supabase immediately — this is a non-negotiable architectural fix. Second, build a Veo 3.1 proof-of-concept that chains 3–4 clips using start/end frame control to validate visual quality and continuity before committing to full development. Third, engage a technology attorney on California AB 723 compliance and AI likeness consent frameworks before writing a single line of product code. The total MVP timeline is **12–16 weeks** with a 4–5 person team, at a base infrastructure cost of roughly **$150/month** plus ~$15 per video generated."
